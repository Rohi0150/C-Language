## 1. Order of Evaluation in Functions and Expressions

In C, the sequence in which the compiler processes **arguments** or **operands** is often **unpredictable and compiler-dependent**.

---

### A. Function Call Arguments

When passing multiple arguments to a function, the compiler may evaluate them from **left-to-right** or **right-to-left**, depending on the specific machine and compiler being used.

#### Program Example — Argument Evaluation:

```c
void test(int x, int y) {
    printf("%d %d", x, y);
}

void main() {
    int a = 10;
    test(++a, a++);
    printf(" %d", a);
}
```

#### Case 1 — Right-to-Left Evaluation:
- `a++` is processed **first** → substitutes `10`, then increments `a` to `11`
- `++a` is processed **next** → increments `11` to `12`, substitutes `12`
- Function receives `(12, 10)`
- **Output:** `12 10 12`

#### Case 2 — Left-to-Right Evaluation:
- `++a` is processed **first** → increments `10` to `11`, substitutes `11`
- `a++` is processed **next** → substitutes `11`, then increments `a` to `12`
- Function receives `(11, 11)`
- **Output:** `11 11 12`

> ⚠️ **Conclusion:** Relying on evaluation order is considered **bad programming practice**. Code should be written to work **independently of the compiler**.

---

### B. Operand Evaluation in Expressions

For standard operators like `+`, the order in which the **left and right operands** are executed is also **compiler-dependent**.

#### Program Example — Operand Execution:

```c
int f(int x) { printf("%d ", x); return x; }
int g(int y) { printf("%d ", y); return y; }

void main() {
    int a = f(5) + g(3);
    printf("%d", a);
}
```

- **Left-to-Right:** Output → `5 3 8`
- **Right-to-Left:** Output → `3 5 8`

> The **final result (sum = 8) is always the same**, but the **order of side effects** (like `printf` statements inside the functions) changes based on the compiler.

## 2. Logical Operators and Short-Circuiting

Unlike standard arithmetic operators, the following operators **always evaluate from left-to-right**:

| Operator | Name |
|----------|------|
| `&&` | Logical AND |
| `\|\|` | Logical OR |
| `? :` | Conditional (Ternary) |
| `,` | Comma |

These operators use **short-circuit logic** to avoid unnecessary execution.

---

### Short-Circuit Rules:

#### Logical OR (`||`):
> If the **first argument is non-zero (true)**, the **second argument is NOT executed** — because the result is already guaranteed to be `true`.

#### Logical AND (`&&`):
> If the **first argument is zero (false)**, the **second argument is NOT executed** — because the result is already guaranteed to be `false`.

---

### Using Short-Circuiting for Flow Control (Without `if-else`)

You can perform conditional logic **without using flow control keywords** like `if` or `else`:

#### Program 1 — Maximum of Two Numbers:

```c
(a > b) && printf("A is big") || printf("B is big");
```

- If `a > b` is **true (1)** → `&&` evaluates `printf("A is big")` → result is non-zero → `||` side is **skipped**
- If `a > b` is **false (0)** → `&&` skips the first print → `||` evaluates `printf("B is big")`

---

#### Program 2 — Even or Odd:

```c
(n % 2) && printf("Odd") || printf("Even");
```

- If `n % 2` is **1 (True)** → prints `"Odd"`
- If `n % 2` is **0 (False)** → skips to `||` and prints `"Even"`

---

#### Program 3 — Positive or Negative:

```c
(n > 0) && printf("Positive") || printf("Negative");
```

- If `n > 0` is **true** → prints `"Positive"`
- If `n > 0` is **false** → prints `"Negative"`

## 3. Compilation Rules and Default Assumptions

- The C compiler reads and processes code **from top-to-bottom**.
- Functions can technically be **defined in any order** in a source file.

### The Default Integer Assumption

If the compiler encounters a **function call before it has seen the function's definition**, it makes the following **default assumptions**:

1. The **return type** is assumed to be `int`
2. The **argument types** are assumed to be `int`

### The Redeclaration Error

If the actual definition of the function appears **later in the file** and does **not match** these assumptions (e.g., the return type is `void`), the compiler throws:

```
Error: Redeclaration to show function
```

#### Program Example — Compilation Error:

```c
void main() {
    show();        // Compiler assumes: int show() — default assumption
}

void show() {      // Error: compiler assumed int, but found void
    printf("hello");
}
```

> The compiler saw `show()` first, assumed it returns `int`. When it later finds `void show()`, it conflicts with the assumption → **Redeclaration Error**.

---

## 4. Three Ways to Fix Declaration Errors

When you call a function **before its definition**, you have **three solutions**:

| # | Solution | Description |
|---|----------|-------------|
| 1 | **Move the Definition** | Place the function definition **above** `main()` or wherever it is first called |
| 2 | **Match the Default** | Change the function's return type to `int` to match the compiler's default assumption |
| 3 | **Use a Function Declaration (Prototype)** | Tell the compiler about the function before calling it |

---

## 5. Function Declarations (Prototypes)

A **function declaration** (also called a **prototype** or **function signature**) informs the compiler about a function's **return type** and **argument types** *before* it is actually called or defined.

### Purpose:
- To **"convince" the compiler** not to use default assumptions
- To allow the compiler to **verify** that function calls match the actual definition

### Syntax:

```c
return_type function_name(type_arg1, type_arg2, ...);
```

Example:
```c
void show(void);
void area(int, float);
```

---

### Key Terminology:

| Term | Meaning |
|------|---------|
| **Declaration / Prototype / Signature** | The line that tells the compiler about a function before it's defined |
| **Declarator** | The **first line** of a function's actual definition |
| **Definition** | The complete function including its body `{}` |

> ⚠️ The **declaration** and the **declarator** must **match exactly** in return type and argument types.

---

### Dummy Names in Declarations:

You can include **parameter names** in a declaration to improve **readability**, but they are **not required** — the compiler only cares about the **types**.

```c
void area(int length, int breadth);  // Dummy names — valid but optional
void area(int, int);                  // Also valid
```

---

### Void Arguments:

If a function takes **no arguments**, it is **good practice** to write `void` inside the parentheses:

```c
void show(void);     // Explicitly shows: takes no arguments
void main(void);     // Same reason why void main(void) is valid
```

---

### Program Example — Using Declarations:

```c
void test(int, float);    // Declaration (prototype) — convinces the compiler

void main() {
    test(5, 5.5);         // Function Call
}

void test(int a, float b) {    // Definition (Declarator must match declaration)
    printf("%d %f", a, b);
}
```

**How it works:**
1. Compiler reads the declaration `void test(int, float);` — now it knows the return type and argument types
2. When it sees `test(5, 5.5)` in `main()`, it verifies against the declaration — no error
3. When it later reaches the actual definition, it matches — no redeclaration error

## 6. Local vs. Global Declarations

The **scope** of a declaration determines which functions can "see" and use it.

| Feature | Local Declaration | Global Declaration |
|---------|-------------------|-------------------|
| **Placement** | Inside a function's `{}` curly braces | Outside all functions (usually at the top of the file) |
| **Scope** | Restricted to **only that specific function** | Visible and accessible to **all functions** in the file |
| **Use Case** | When a function is called by only one other function | When a function is called by many functions — avoids redundant declarations |

---

### Program Example — Power of Global Declaration:

```c
void f3(void);    // Global Declaration — known to ALL functions below

void f1() { f3(); }
void f2() { f3(); }
void main() { f3(); }

void f3() { printf("hello f3"); }
```

> If `f3()` is called by **10,000 different functions**, placing **one global declaration** at the top is far more efficient than writing **10,000 local declarations** inside each function separately.

---

### Local Declaration Example:

```c
void main() {
    void f3(void);   // Local Declaration — only main() knows about f3
    f3();
}

void f1() {
    // f3 is NOT visible here — would cause an error
}
```

---

## 7. Important Notes & Best Practices

- ✅ Always write function **prototypes/declarations** at the top of your file or in a header file
- ✅ Use `void` explicitly in function parameters when a function takes no arguments: `void show(void)`
- ✅ The **declaration** is only for the **compiler** — it does not execute any logic
- ✅ Actual **execution logic** resides entirely in the **function definition**
- ❌ Never rely on **argument evaluation order** in function calls (e.g., `test(++a, a++)`) — it's compiler-dependent and bad practice
- ❌ Never rely on **operand evaluation order** in expressions when side effects are involved
- ✅ Logical operators (`&&`, `||`) **always** evaluate left-to-right and can be used creatively for flow control
